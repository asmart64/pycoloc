
Is the Costes thresholding algorithm symmetric with respect to a swapping of Red and green images. I.E. do we find the swapped thresholds?
## Symmetry of the Costes Autothreshold

This is a subtle and important question. The short answer is: **not exactly**, and the asymmetry comes from two sources.

---

### Source 1: The Regression Line

The threshold pair $(T_R, T_G)$ is constrained to lie on the regression line fitted to the scatter plot:

$$G = a \cdot R + b$$

If you swap channels, the regression is now:

$$R = a' \cdot G + b'$$

These are **not the same line**. Ordinary least squares minimizes residuals in the *dependent* variable only — the line $G = aR + b$ minimizes vertical residuals, while $R = a'G + b'$ minimizes horizontal residuals. They coincide only when $r = \pm 1$ (perfect correlation). In general:

$$a' \neq \frac{1}{a} \quad \text{and} \quad b' \neq -\frac{b}{a}$$

So the path along which the algorithm walks when scanning thresholds is geometrically different depending on which channel is treated as dependent. The threshold pairs found will generally not be exact swaps of each other.

---

### Source 2: The Pearson Criterion Is Symmetric, But the Pixel Set Is Not

The stopping criterion — Pearson $r$ of sub-threshold pixels $\leq 0$ — uses Pearson's $r$, which is itself symmetric in $R$ and $G$. So this part does not introduce asymmetry.

However, the **sub-threshold pixel set** $S$ is defined as:

$$S = \{ i : R_i < T_R \;\text{and}\; G_i < T_G \}$$

This set is the same regardless of which channel you call "red" or "green" — it depends only on the pair $(T_R, T_G)$. So the Pearson computation on $S$ is symmetric.

The asymmetry arises entirely from the regression step determining **which $(T_R, T_G)$ pairs are visited** during the scan.

---

### What Happens in Practice

Consider the scatter plot geometry:

```
G ↑         
  |    . cloud
  |   .
b |  .
  | .
  +----------→ R
```

- Walking along $G = aR + b$ from high $R$ downward visits one set of $(T_R, T_G)$ pairs
- Walking along $R = a'G + b'$ from high $G$ downward visits a different set

The threshold found by the first walk and the threshold found by the second will generally **not satisfy** $T_G^{(1)} = T_R^{(2)}$ and $T_R^{(1)} = T_G^{(2)}$ unless the scatter is perfectly symmetric around the diagonal.

---

### The Special Case Where Symmetry Holds

Exact symmetry is recovered when:

- The scatter plot is symmetric about the $R = G$ diagonal, i.e. the two channels have the same mean, variance, and joint distribution
- The regression line passes through the origin with slope $a = 1$ and $b = 0$

In this case both regression lines are identical ($G = R$) and the algorithm finds exactly swapped thresholds.

---

### Practical Consequence

In real data, the two channels almost never have matched means and variances — one fluorophore is brighter, one antibody labels more densely, etc. So if you run Coloc 2 with channels swapped, you will in general get:

- Different $T_R$, $T_G$ values
- Slightly different M₁ and M₂ (because the thresholds change)
- The same Pearson $r$ and ICQ (which don't depend on the threshold)

This is actually an underappreciated source of variability in published colocalization results. **The convention in Coloc 2 is to always put the dimmer or sparser channel as channel 1** (red), which tends to give a more stable regression, but this is a convention, not a mathematical fix.

The asymmetry is a real limitation of the Costes method — it is one of the motivations for methods that use orthogonal regression (minimizing perpendicular residuals) rather than ordinary least squares, which would be fully symmetric by construction.

I’ll break down each patch as an exact before/after behavior change and tie it to the Coloc2 mechanism it mirrors, with precise file locations.**Exact Patch Breakdown (Coloc2 Pairing Changes)**

1. Regression mode default changed  
Location: colocalization_app.py

- Before: orthogonal regression checkbox defaulted to False.
- After: default is True.
- Effect: analysis now starts in orthogonal/TLS mode unless you disable it. This changes fitted slope/intercept and therefore paired thresholds.

2. Costes method description updated to Coloc2-style semantics  
Location: colocalization_app.py

- Before docs described:
  - threshold scan from Ch1 max to min,
  - background mask with AND,
  - stop at first r ≤ 0 with foreground support constraint.
- After docs describe:
  - 1-intensity stepper logic,
  - step axis chosen by slope magnitude,
  - background mask with OR,
  - Coloc2-like stop criteria (tiny r, NaN, or r rebound).

3. Removed fixed linspace candidate sweep  
Location: colocalization_app.py

- Before: built candidates with linspace(max1, min1, n_steps+1).
- After: no linspace sweep; iterative stepper decreases by 1 each iteration.
- Effect: search path now matches Coloc2 SimpleStepper behavior.

4. Added channel-dependent stepping axis  
Location: colocalization_app.py

- Before: always scanned T1 and derived T2 from regression line.
- After:
  - if -1 < slope < 1, step on Ch1;
  - otherwise step on Ch2 and derive paired Ch1 threshold by inverse mapping.
- Effect: reproduces Coloc2 strategy for steep regressions.

5. Threshold rounding changed to integer-level image thresholds  
Location: colocalization_app.py

- Before: thresholds were floating candidates from linspace mapping.
- After: thresholds rounded with floor(x + 0.5), then clipped to image min/max.
- Effect: candidate thresholds align to integer intensity bins as in Coloc2.

6. Background mask changed from AND to OR  
Location: colocalization_app.py

- Before: background pixels were c1 < T1 AND c2 < T2.
- After: background pixels are c1 < T1 OR c2 < T2.
- Effect: this is the major behavioral change; background set is larger and typically changes r-vs-threshold trajectory significantly.

7. Minimal pixel validity and finite-r handling changed  
Location: colocalization_app.py

- Before: skipped candidates with too few pixels/zero std and continued loop.
- After: computes r only if mask has at least 3 pixels and both std are valid, else sets r = NaN.
- Effect: integrates invalid states into stop logic instead of silently skipping.

8. Removed foreground support guardrails  
Location: previous logic around old min_fg_pixels block replaced in colocalization_app.py

- Before: required minimum foreground support and per-channel support; used these guards to accept/stop.
- After: those constraints are removed.
- Effect: threshold can be accepted even when surviving foreground is small, matching Coloc2 closer but less conservative.

9. Stop condition replaced with Coloc2-like termination  
Location: colocalization_app.py

- Before: stop when r ≤ 0 under support constraints.
- After stop when any is true:
  - r non-finite,
  - next threshold < 1,
  - r < 1e-4,
  - r > last_r (rebound).
- Effect: emulates Coloc2 stepper stopping rather than first zero-crossing logic.

10. Fallback policy changed  
Location: colocalization_app.py

- Before: tracked best supported candidate; fallback to global minimum-r candidate when needed.
- After: keeps current stepper output unless all r are non-finite; then fallback to conservative defaults.
- Effect: aligns with stepper-driven output model.

11. Minor signature/variable cleanup  
Location: colocalization_app.py

- n_steps remains in signature but is no longer used by the new stepper loop.
- max2/min2 handling expanded for clipping both thresholds.
- Effect: compatibility retained, behavior shifted to stepper model.

**Summary of what changed mathematically**

- Regression line source changed by default (orthogonal on).
- Threshold traversal changed (unit-step, slope-dependent axis).
- Background definition changed (OR instead of AND).
- Stop rule changed (stepper criteria vs first r ≤ 0).
- Conservative robustness guards were removed.

These are exactly the reasons you can see equal-looking thresholds but different biological-looking output patterns in M1/M2/tM1/tM2.
